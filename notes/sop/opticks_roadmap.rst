notes/sop/opticks_roadmap
===========================


Major Issue 1 : Coping with continous JUNOSW simulation evolution in geometry and model
-----------------------------------------------------------------------------------------

* some CPU side changes require custom development to reproduce GPU side
* any change can break correctness
* any change can break performance

Current use of gitlab-ci is limited to CPU only testing of build and geometry conversion.
A gitlab-runner installed on Ada GPU workstation has succeeded to perform tasks
coordinated by the gitlab-ci instance.

Problems with this - is that the free tier of gitlab has limited reporting capabilities.
Workaround is to use gitlab-ci to give pass/fail status with links smuggled in to
external html pages that are generated by monitoring tools, eg: Prometheus + Graphana.

Handling the issue of continuous change can be split into the below two tasks
of "OpticksNightly" and "OpticksMonitoring".


"OpticksNightly" : Expand use of gitlab-ci using GPU workstation gitlab-runner - for small(non-batch) tests run nightly
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

* expand use of gitlab-ci for opticks
* expand use of gitlab-ci for opticks+JUNOSW

Tasks:

* work out how to use gitlab-ci junit testing and reporting
* develop device suite of short[1] validation and performance tests
* work with the "OpticksMonitoring" reponsible to integrate links to external monitoring tool

[1] Goal is nightly running : full suite should complete on Ada GPU workstation within 4 hours

An "OpticksWeekly" is also possible for longer tests.


"OpticksMonitoring" : Adopt Industry standard Monitoring tools eg Prometheus + Graphana
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Validation and testing must become a continuous activity with metrics
recording and historical monitoring, eg to identify:

0. validation failures in comparison with Geant4
1. performance degradation
2. VRAM usage increases

* investigate Prometheus + Graphana, are they the most appropriate open source tools for the task
* devise monitoring pipeline for Opticks

  * natural to extend sysrap/tests/sreport.cc reports [NP.hh/NPFold.h serializations] to have json summary
  * why use json ? its a standard - so can use standard tools
  * then C++ ingester using sqlite3 C API to construct sqlite DB from the json summary folders, and referenced arrays (NPFold.h/NP.hh)
  * it makes no sense to shoehorn arrays into json (NP machinery handles arrays better than json)

* investigate how to integrate the reporting tool with gitlab-ci coordination

Question for AI: Does gitlab-ci free tier allow links to be included in junit or other reports
that can reference external presentation of more detailed metrics ?


* initial thoughts on pipeline mechanics ~/j/opticks_monitoring

  * job summaries[json+NPFold] ingested into sqlite3 DB using C api (so can use NPFold.h/NP.hh)
  * ~/j/opticks_monitoring
  * https://code.ihep.ac.cn/blyth/j/-/tree/main/opticks_monitoring


Major Issue 2 : Not enough NVIDIA GPUs are available
--------------------------------------------------------

To handle this issue:

1. make better use of the NVIDIA GPUs that are available with "OpticksServer" and "OpticksOptimization" tasks
2. generalize Opticks to work with non-NVIDIA GPUs with "OpticksWithoutOptiX" task


"OpticksServer" : prototype is FastAPI+nanobind server, try instead NVIDIA Triton Inference Server
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Aim for high GPU utilization by sharing GPUs between CPU nodes.

Requires development of interface "glue" that makes Opticks fulfil
the API required to become a Custom C++ backend allowing it to be used from
NVIDIA Triton Inference Server.


"OpticksOptimization" : Use AI code analysis/generation to optimize Opticks
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

AI code analysis and generation has become sufficient to realistically aid optimization.
Critical parts of Opticks code can be reviewed and optimized after ensuring unit tests
are sufficient to ensure correctness is maintained and monitoring of performance
and VRAM usage is in place.

Areas likely to yield the greatest benefits:

* ray intersection with various simple solid shapes
* ray intersection with compound solid shapes (CSG) and the tightly coupled representation of CSG trees

Also AI can be used to assist with using NVIDIA Nsight Compute performance tools
to optimize Opticks kernels There are likely to be low-hanging fruits as AI can
give a kickstart on how to use the tools effectively.


"OpticksWithoutOptiX" : Investigate alternatives to NVIDIA OptiX ray tracing
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Opticks structural generalization to work with multiple backend ray trace engines.

Use AI to assist with prototyping alternative ray tracing approaches:

* Vulkan Ray Tracing - works on any modern GPU
* Apple Metal Ray Tracing - Apple laptops are widely available
* Intel Embree - CPU ray tracing
* develop pure CUDA acceleration structure (eg BVH) and traversal based on open source examples

  * prototype for GPU generalization
